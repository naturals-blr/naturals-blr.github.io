Local Dev Setup — Making & Pushing Changes

Steps to clone, edit, and push code changes to the naturals-blr/naturals-ops repository.

1. Clone the repo

git clone https://github.com/naturals-blr/naturals-ops.git
cd naturals-ops

2. Create a GitHub PAT (classic)

  1. Go to GitHub Settings → Developer Settings → Personal Access Tokens → Tokens (classic)
  2. Generate a new classic token with scope repo (full control of private repositories)
  3. Copy the token — it looks like github_pat_11B6YBCYQ...
  4. Save it somewhere safe (you won't see it again)

3. Set remote URL with PAT

Replace YOUR_PAT_HERE with your actual PAT and run:

git remote set-url origin https://YOUR_PAT_HERE@github.com/naturals-blr/naturals-ops.git

Example (with a sample PAT):

# Replace with your own PAT
git remote set-url origin https://YOUR_GITHUB_PAT@github.com/naturals-blr/naturals-ops.git

4. Make changes & push

  1. Edit files with your editor of choice
  2. Stage and commit: git add . && git commit -m "your message"
  3. Push: git push (authenticates via the PAT in the remote URL)
  4. Push triggers naturals-ops.yml — build + deploy to Netlify and GitHub Pages
💡 Local Dev Tips
  • Always pull latest before editing: git pull
  • Run npm run export:build-data && python3 build/build.py before testing locally to get latest data
  • Test admin changes locally: npm start, open http://localhost:3456
  • If you create a new GitHub Secret, add it to .env locally and reference it in naturals-ops.yml
  • The PAT in the remote URL can be rotated anytime — just re-run git remote set-url origin

Token Management

The application uses multiple tokens for different services. Below is the complete reference for obtaining, configuring, and rotating each token.

1 Facebook / Meta Page Access Tokens

Per-store tokens stored in Google Sheet (fb_page_token column), synced to build/data/stores.json. Used for Facebook & Instagram publishing. Never expire unless Facebook password changes.

Generate Short-Lived User Token

  1. Open Meta Graph API Explorer
  2. Meta App dropdown (top-right): select Naturals Social Automation
  3. User or Page dropdown: set to User Token
  4. Add permissions: pages_show_list, pages_manage_posts, pages_read_engagement, business_management
  5. Generate token, approve all 5 salon locations

Extend & Extract Permanent Tokens

  1. Copy user token → paste into Access Token Debugger → click Extend Access Token
  2. Paste the 60-day token back into Graph API Explorer
  3. Query: me/accounts?fields=name,access_token,id
  4. Extract 5 permanent page tokens (Expires: Never) from JSON response
  5. Save as fb_page_tokens_backup.json in a secure location (not in repo)

Update & Verify

  1. Open the master Stores Configuration Google Sheet
  2. Paste each token into its store's fb_page_token column

2 Google OAuth (Refresh Token + Client ID/Secret)

Used for Google Business Profile API (reviews), Google Sheets API (data sync), and social publishing to Google. Tokens expire every ~7 days and need re-authorization via the discovery script.

Setup

  1. Go to Google Cloud Console → APIs & Services → Credentials
  2. Create OAuth 2.0 Client ID (Desktop app type)
  3. Download as google_oauth_credentials.json or client_secret_*.json
  4. Run: DISPLAY_AUTH_URL=true node scripts/test/discoverGoogleAccounts.js
  5. Visit the printed auth URL, authorize, copy the refresh token from oauth-tokens.json

Used by the admin panel "Publish via Google" path. A Google Apps Script Web App that receives campaign payloads, stores images in Drive, and fires a repository_dispatch to trigger the publish workflow.

Setup

  5. Copy the Web App URL → save as GitHub secret: GAS_WEBHOOK_URL
💡 GAS Script Tips

  • Deploy as new version whenever the script is updated
  • Test with a GET to the Web App URL — should return {"ok":true,"message":"GAS Relay is running"}

🔒 Security Gate: INTERNAL_GATEKEEPER_KEY

This is a shared cryptographic password (a long string of random numbers and letters) that acts like a private secure handshake between your Google Apps Script relay and your GitHub repository.

Because the GitHub workflows are locked down to block outsiders, GitHub will instantly reject any repository_dispatch command unless it arrives containing this exact secret string in the auth_token field of the client_payload. If the token is missing or wrong, the workflow exits immediately with UNAUTHORIZED DISPATCH ATTEMPT DETECTED.

Step 1: Generate a Secure Key

Create a completely random 32-character secure key by running this terminal command: 

openssl rand -hex 32

Alternatively, use a secure password generator to create a strong, random 32-character string. It will look something like: 

f3bc7a12d98e4c5b6a7f8e9102c3d4e5

Step 2: Paste it into GitHub Secrets

This tells GitHub what password to expect when a repository_dispatch arrives.

  1. Go to your GitHub repository naturals-ops.
  2. Click Settings → Secrets and variables (left menu) → Actions.
  3. Click the green New repository secret button.
  4. Set the Name to exactly: INTERNAL_GATEKEEPER_KEY
  5. Paste your generated random string into the Value box.
  6. Click Add secret.

Step 3: Paste it into Google Apps Script

This tells your automated relay script to send that password along with every campaign payload.

  1. Open your Google Apps Script project.
  2. Click the gear icon (Project Settings) on the left side panel.
  3. Scroll down to Script Properties and click Edit Script Properties.
  4. Click Add property.
  5. Set the Property name to exactly: INTERNAL_GATEKEEPER_KEY
  6. Paste the exact same random string into the Value box.
  7. Click Save script properties.

Once both systems are saving the exact same key string, your Apps Script relay can seamlessly pass through the GitHub security gate without any manual intervention.
